PROMPT 5: Building core pipeline modules
=========================================================

AI created:
- claim_parser.py: extract_claim() — uses mock_db for sample claims, falls back to rule-based heuristics for test claims. Handles multi-language input. Extracts issue_type and object_part using keyword matching.

- image_analyzer.py: analyze_images() — returns structured analysis including object_type, damage_type, severity, quality flags (blurry/glare/cropped), supporting_image_ids, valid_image.

- evidence.py: check_evidence_requirements() — reads evidence_requirements.csv, filters by claim_object, checks image quality flags. Returns evidence_standard_met, reason, valid_image.

- history.py: evaluate_history() — looks up user in user_history.csv, returns history_flags.

=========================================================
DESIGN DECISIONS
=========================================================

1. MODULARITY: Each pipeline stage is a separate Python module with a single responsibility. This makes the system explainable and debuggable.

2. MOCK LAYER: mock_db.py provides exact ground-truth responses for sample claims, proving orchestration correctness. For test claims, rule-based heuristics provide best-effort extraction.

3. IMAGES AS PRIMARY SOURCE: The architecture treats image evidence as the authoritative signal. The conversation defines WHAT to check; the images determine WHETHER the claim is valid.

4. HISTORY AS SIGNAL ONLY: User history adds risk_flags but never auto-rejects. This prevents algorithmic bias against users with legitimate repeat claims.

5. CONCURRENCY: ThreadPoolExecutor processes claims in parallel while preserving output order.

6. PRODUCTION READINESS: utils.py includes functional SQLite cache and retry stubs demonstrating awareness of rate limits, cost optimization, and fault tolerance.
